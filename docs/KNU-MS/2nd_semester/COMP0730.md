### 1. Prioritise tests to maximise early unique line coverage.

Coverage Matrix:

|     | S1  | S2  | S3  | S4  | S5  |
| --- | --- | --- | --- | --- | --- |
| T1  | 1   | 1   | 0   | 0   | 0   |
| T2  | 0   | 1   | 1   | 0   | 0   |
| T3  | 0   | 0   | 1   | 1   | 0   |
| T4  | 1   | 0   | 0   | 1   | 1   |

- Tasks:
  - A. Define the Representation of a GA individual.
  - B. Define a fitness that rewards earlier unique coverage(earlier in a "collective" sense)
  - C. Give one crossover and one mutation operator for your representation.
  - D. With population size=2, parents $P_1=T1\to T2\to T3\to T4, P_2=T4\to T3\to T2\to T1$ Compute each order's coverage progression and your fitness. Create two plausible offspring and estimate their fitness.
  - E. Pcik the next generation (size=2) under: i)elitism, ii) generational replacement, iii) gradual replacement (top k). State the average fitness for each.

#### Answer

GA를 이용한 Early unique line coverage를 최대화하는 테스트 실행 순서 찾는 문제
데이터:

- Statements: \{S1, S2, S3, S4, S5\}
- T1: \{S1, S2\}
- T2: \{S2, S3\}
- T3: \{S3, S4\}
- T4: \{S1, S4, S5\}
  A. GA 개체(Individual)의 표현
- 해석: GA가 탐색할 해(solution)를 유전형(genotype)으로 어떻게 표현할지 정의하는 문제
- 풀이: 이 문제는 테스트 케이스 T1, T2, T3, T4의 순서를 찾는 문제
  개체(individual)는 테스트 케이스의 순열(permutation)로 표현 가능 - 예시: $(T1, T2, T3, T4)$ 또는 $(T4, T3, T2, T1)$
  B. 적합도(Fitness) 함수
- 해석: 초기에 고유한 커버리지를 달성하는 순서에 높은 점수를 주는 적합도 함수 정의 문제
- 풀이: 4번 문제에서 정의된 조기 커버리지 점수(Early coverage score) 5를 사용
  이 점수는 각 테스트 실행 후 누적된 고유 라인 커버리지 비율의 평균 - 계산식: $Fitness = \frac{1}{n} \sum_{k=1}^{n} C_k$ - $n$: 총 테스트 케이스 수 (여기서는 4) - $C_k$: $k$번째 테스트까지 실행했을 때의 누적 고유 라인 커버리지 비율 ( (누적 고유 라인 수) / (총 고유 라인 수 5) ) - 이 점수가 높을수록 초반에 더 많은 라인을 커버한다는 의미
  C. 교배(Crossover) 및 돌연변이(Mutation) 연산자
- 해석: A에서 정의한 순열 표현에 적합한 교배 연산자 1개와 돌연변이 연산자 1개를 제시하는 문제
- 풀이: - 교배 (Crossover): 순서 기반 표현이므로 순서 교배(Order Crossover, OX) 또는 부분 매핑 교배(Partially Mapped Crossover, PMX)를 사용 가능 - 돌연변이 (Mutation): 교환 돌연변이(Swap Mutation)를 사용 가능
  (예: 순열에서 임의의 두 테스트 위치를 서로 맞바꿈. $T1 \rightarrow T2 \rightarrow T3 \rightarrow T4 \Longrightarrow T1 \rightarrow T4 \rightarrow T3 \rightarrow T2$)
  D. 부모 및 자손 적합도 계산
- 부모 $P_1$ = $T1 \rightarrow T2 \rightarrow T3 \rightarrow T4$
  1. T1: \{S1, S2\}. 누적: \{S1, S2\}. (2/5 = 0.4)
  2. T2: \{S2, S3\}. 누적: \{S1, S2, S3\}. (3/5 = 0.6)
  3. T3: \{S3, S4\}. 누적: \{S1, S2, S3, S4\}. (4/5 = 0.8)
  4. T4: \{S1, S4, S5\}. 누적: \{S1, S2, S3, S4, S5\}. (5/5 = 1.0)
  - $P_1$ 적합도: $(0.4 + 0.6 + 0.8 + 1.0) / 4 = 2.8 / 4 = \mathbf{0.7}$
- 부모 $P_2$ = $T4 \rightarrow T3 \rightarrow T2 \rightarrow T1$
  1. T4: \{S1, S4, S5\}. 누적: \{S1, S4, S5\}. (3/5 = 0.6)
  2. T3: \{S3, S4\}. 누적: \{S1, S3, S4, S5\}. (4/5 = 0.8)
  3. T2: \{S2, S3\}. 누적: \{S1, S2, S3, S4, S5\}. (5/5 = 1.0)
  4. T1: \{S1, S2\}. 누적: \{S1, S2, S3, S4, S5\}. (5/5 = 1.0)
  - $P_2$ 적합도: $(0.6 + 0.8 + 1.0 + 1.0) / 4 = 3.4 / 4 = \mathbf{0.85}$
- 그럴듯한 자손 2개 생성 및 적합도 추정 - 자손 1 ($O_1$): $P_1$에 Swap(T2, T4) 돌연변이 적용 - $O_1 = T1 \rightarrow T4 \rightarrow T3 \rightarrow T2$ 1. T1: \{S1, S2\}. 누적: \{S1, S2\}. (2/5 = 0.4) 2. T4: \{S1, S4, S5\}. 누적: \{S1, S2, S4, S5\}. (4/5 = 0.8) 3. T3: \{S3, S4\}. 누적: \{S1, S2, S3, S4, S5\}. (5/5 = 1.0) 4. T2: \{S2, S3\}. 누적: \{S1, S2, S3, S4, S5\}. (5/5 = 1.0) - $O_1$ 적합도: $(0.4 + 0.8 + 1.0 + 1.0) / 4 = 3.2 / 4 = \mathbf{0.8}$ - 자손 2 ($O_2$): $P_2$에 Swap(T3, T2) 돌연변이 적용 - $O_2 = T4 \rightarrow T2 \rightarrow T3 \rightarrow T1$ 1. T4: \{S1, S4, S5\}. 누적: \{S1, S4, S5\}. (3/5 = 0.6) 2. T2: \{S2, S3\}. 누적: \{S1, S2, S3, S4, S5\}. (5/5 = 1.0) 3. T3: \{S3, S4\}. 누적: \{S1, S2, S3, S4, S5\}. (5/5 = 1.0) 4. T1: \{S1, S2\}. 누적: \{S1, S2, S3, S4, S5\}. (5/5 = 1.0) - $O_2$ 적합도: $(0.6 + 1.0 + 1.0 + 1.0) / 4 = 3.6 / 4 = \mathbf{0.9}$
  E. 다음 세대 선정 (크기=2)
- 현재 풀(Pool): $P_1(0.7)$, $P_2(0.85)$, $O_1(0.8)$, $O_2(0.9)$
- 적합도 순위: $O_2(0.9) > P_2(0.85) > O_1(0.8) > P_1(0.7)$
- i) 엘리트주의 (Elitism):
  - 방식: 부모와 자손을 합친 전체 풀에서 적합도가 가장 높은 2개를 선택
  - 다음 세대: \{ $O_2$, $P_2$ \}
  - 평균 적합도: $(0.9 + 0.85) / 2 = \mathbf{0.875}$
- ii) 세대 교체 (Generational replacement):
  - 방식: 부모 세대($P_1, P_2$)를 자손 세대($O_1, O_2$)로 완전히 교체
  - 다음 세대: \{ $O_1$, $O_2$ \}
  - 평균 적합도: $(0.8 + 0.9) / 2 = \mathbf{0.85}$
- iii) 점진적 교체 (Gradual replacement, top k):
  - 방식: "top k" 12는 보통 전체 풀에서 상위 k개(여기서는 k=2)를 선택하는 것을 의미하며, 이는 (i) 엘리트주의와 동일
  - 다음 세대: \{ $O_2$, $P_2$ \}
  - 평균 적합도: $(0.9 + 0.85) / 2 = \mathbf{0.875}$

### 2. APR via GA

GA를 이용한 자동 프로그램 수정(APR) 문제.

Buggy code:

```python
def shipping_fee(weight_kg):
 # policy: fee is 0 when weight $\le5$, otherwise 10
 if weight_kg < 5: #faulty operator; should be <=
  return 0
 else:
  return 10
```

Test suite (oracle):

- T1: `shipping_fee(0) → 0`
- T2: `shipping_fee(5) → 0`
- T3: `shipping_fee(5.0) → 0`
- T4: `shipping_fee(6) → 10`

A. Define a genotype with a minimal set of editable choices, and explain how it maps to the phenotype (the concrete program).

- 해석: 버그 수정을 위한 "최소한의 편집 선택"을 유전형(genotype)으로 정의하고, 이것이 실제 코드(phenotype)로 어떻게 매핑되는지 설명하는 문제.
- 풀이:
  - 유전형 (Genotype): 버그가 있는 18번 라인의 비교 연산자 (`<`)를 대체할 연산자를 나타내는 정수(Integer)이다.
  - 표현형 (Phenotype): 유전형에 따라 연산자가 수정된 실제 Python 코드이다.
  - 매핑 (Mapping):
    - 1 $\rightarrow$ `if weight_kg < 5:`
    - 2 $\rightarrow$ `if weight_kg <= 5:`
    - 3 $\rightarrow$ `if weight_kg > 5:`
    - 4 $\rightarrow$ `if weight_kg >= 5:`
    - 5 $\rightarrow$ `if weight_kg == 5:`
    - 6 $\rightarrow$ `if weight_kg != 5:`

B. Specify one mutation operator and (if applicable) one crossover operator for your representation.

- 해석: A에서 정의한 표현형에 맞는 돌연변이 연산자 1개와 (적용 가능하다면) 교배 연산자 1개를 지정하는 문제.
- 풀이:
  - 돌연변이 (Mutation): 무작위 리셋(Random Resetting). 현재 유전형(예: 1)을 다른 가능한 정수(예: 2~6 중 하나)로 무작위 변경한다.
  - 교배 (Crossover): 이 유전형은 단일 유전자(정수 하나)로 구성되어 교배가 큰 의미가 없다. (만약 유전형이 `[연산자, 값]`처럼 여러 유전자로 구성되었다면, 균등 교배(Uniform Crossover)를 사용할 수 있다.)

C. Define a fitness based on the test suite above.

- 해석: 테스트 스위트를 기반으로 적합도를 정의하는 문제.
- 풀이: (통과한 테스트 케이스 수) / (전체 테스트 케이스 수)
  - $Fitness = \frac{\text{Pass(T1)} + \text{Pass(T2)} + \text{Pass(T3)} + \text{Pass(T4)}}{4}$
  - (여기서 Pass(T)는 통과 시 1, 실패 시 0이다.)

D. Compute fitness for these two candidate programs:

- 해석: $C_1$(버그 버전)과 $C_2$(패치 버전)의 적합도를 C에서 정의한 함수로 계산한다.
- 풀이:
  - $C_1$: `if weight_kg < 5:` (버그 버전)
    - T1(0): `0 < 5` (True) $\rightarrow$ 0. (통과)
    - T2(5): `5 < 5` (False) $\rightarrow$ 10. (실패, 0 기대)
    - T3(5.0): `5.0 < 5` (False) $\rightarrow$ 10. (실패, 0 기대)
    - T4(6): `6 < 5` (False) $\rightarrow$ 10. (통과)
    - $C_1$ 적합도: (1 + 0 + 0 + 1) / 4 = 2 / 4 = 0.5
  - $C_2$: `if weight_kg <= 5:` (패치 버전)
    - T1(0): `0 <= 5` (True) $\rightarrow$ 0. (통과)
    - T2(5): `5 <= 5` (True) $\rightarrow$ 0. (통과)
    - T3(5.0): `5.0 <= 5` (True) $\rightarrow$ 0. (통과)
    - T4(6): `6 <= 5` (False) $\rightarrow$ 10. (통과)
    - $C_2$ 적합도: (1 + 1 + 1 + 1) / 4 = 4 / 4 = 1.0

E. If your population contains only $C_1$ and $C_2$, which next-generation policy would you prefer and why?

- 해석: 현재 인구(population)가 $C_1$(적합도 0.5)과 $C_2$(적합도 1.0)만으로 구성될 때, 어떤 세대 교체 정책을 선호하는지 묻는 문제.
- 풀이:
  - 선호 정책: 엘리트주의(Elitism) 또는 점진적 교체(Gradual replacement (top k))
  - 이유: 현재 인구에 완벽한 해(적합도 1.0)인 $C_2$가 이미 존재한다. 엘리트주의는 현 세대의 가장 우수한 개체를 다음 세대로 무조건 생존시키므로, 이 완벽한 해($C_2$)를 잃어버릴 위험이 없다. 반면, 세대 교체(Generational replacement)는 모든 부모($C_1, C_2$)를 자손으로 대체하는데, 이 과정에서 완벽한 해가 소실될 위험이 있다.

### 3. Coverage Computation

주어진 코드와 테스트 스위트에 대한 커버리지를 계산하는 문제.

```python
def f(a, b, c):
 if a and b: # D1
  S2: $x=1$
 else:
  S3: $x=2$
 if c: # D2
  S5: $y=x+1$
 else:
  S6: $y=x-1$
 S7: return y
```

Test suite:

- T1: `(a=True, b=False, c=True)`
- T2: `(a=True, b=True, c=False)`

A. For T1 and T2, list the statements executed and compute overall statement coverage.

- 해석: T1과 T2가 실행하는 구문(S)을 나열하고, 전체 구문 커버리지를 계산한다.
- 풀이:
  - T1 실행 구문:
    - `if True and False:` (False) $\rightarrow$ else 실행
    - S3
    - `if True:` (True)
    - S5
    - S7
    - 실행된 구문: \{S3, S5, S7\}
  - T2 실행 구문:
    - `if True and True:` (True)
    - S2
    - `if False:` (False) $\rightarrow$ else 실행
    - S6
    - S7
    - 실행된 구문: \{S2, S6, S7\}
  - 전체 구문 커버리지:
    - 전체 구문: \{S2, S3, S5, S6, S7\} (총 5개)
    - 실행된 구문: \{S2, S3, S5, S6, S7\} (5개 모두 실행)
    - 커버리지: 5 / 5 = 100%

B. Compute decision (branch) coverage for D1 and D2.

- 해석: D1(`a and b`)과 D2(`c`)의 결정(분기) 커버리지를 계산한다.
- 풀이:
  - D1 (`if a and b`):
    - T1: `a and b` = False (Else 브랜치 커버)
    - T2: `a and b` = True (If 브랜치 커버)
    - D1 커버리지: 2 / 2 = 100%
  - D2 (`if c`):
    - T1: `c` = True (If 브랜치 커버)
    - T2: `c` = False (Else 브랜치 커버)
    - D2 커버리지: 2 / 2 = 100%

C. Compute condition coverage for a, b, and c.

- 해석: 개별 조건 $a, b, c$ 각각의 T/F 커버리지를 계산한다.
- 풀이:
  - 전체 조건: $a, b, c$
  - 조건 'a':
    - T1: $a = \text{True}$
    - T2: $a = \text{True}$
    - 결과: True만 커버, False는 커버 안 됨.
    - 'a' 커버리지: 1 / 2 = 50%
  - 조건 'b':
    - T1: $b = \text{False}$
    - T2: $b = \text{True}$
    - 결과: True, False 모두 커버.
    - 'b' 커버리지: 2 / 2 = 100%
  - 조건 'c':
    - T1: $c = \text{True}$
    - T2: $c = \text{False}$
    - 결과: True, False 모두 커버.
    - 'c' 커버리지: 2 / 2 = 100%

D. Is the current test suite MC/DC-adequate for D1?

- 해석: D1 (`if a and b`)에 대해 MC/DC가 충족되었는지 묻는 문제. MC/DC는 각 개별 조건(a, b)이 다른 조건은 고정한 채 자신(T/F)만 변경되어 전체 결정(D1)의 결과(T/F)를 바꾸는 테스트 쌍이 존재하는지를 본다.
- 풀이: 아니다.
  - 'b'에 대한 독립적 영향:
    - T2: ($a=\text{True}, b=\text{True}$) $\rightarrow$ D1 = True
    - T1: ($a=\text{True}, b=\text{False}$) $\rightarrow$ D1 = False
    - 'a'가 True로 고정된 상태에서 'b'가 T/F로 바뀌자 D1이 T/F로 바뀌었다. ('b'는 MC/DC 만족)
  - 'a'에 대한 독립적 영향:
    - T2: ($a=\text{True}, b=\text{True}$) $\rightarrow$ D1 = True
    - (필요한 테스트): ($a=\text{False}, b=\text{True}$) $\rightarrow$ D1 = False
    - 'b'가 True로 고정된 상태에서 'a'가 T/F로 바뀌는 쌍이 필요하다. 하지만 현재 테스트 스위트에는 ($a=\text{False}$)인 경우가 없다.
    - 결론: 'a'에 대한 MC/DC 쌍이 없으므로 D1은 MC/DC를 충족하지 못한다.

### 4. Coverage and Fault Detection

두 가지 테스트 순서 $O_A$와 $O_B$를 '조기 커버리지 점수'와 'APFD' (여기서는 APFD'로 정의됨) 관점에서 비교하는 문제.

Coverage Matrix (S1-S5, 총 5개):

- T1: \{S1, S2\}
- T2: \{S2, S3\}
- T3: \{S3, S4\}
- T4: \{S1, S4, S5\}

Fault Detection Matrix (F1-F2, 총 2개):

- T1: \{\}
- T2: \{F1\}
- T3: \{F2\}
- T4: \{\}

A. For $O_A$ compute the cumulative unique coverage after each position and the early coverage score. Do the same for $O_B$.

- 해석: $O_A$와 $O_B$ 각각에 대해 "각 테스트 실행 후 누적된 고유 라인 커버리지 비율의 평균" (조기 커버리지 점수)을 계산한다.
- 풀이:
  - 순서 $O_A$: \[T4, T1, T2, T3\]
    1. T4: \{S1, S4, S5\}. 누적: \{S1, S4, S5\}. (3/5 = 0.6)
    2. T1: \{S1, S2\}. 누적: \{S1, S2, S4, S5\}. (4/5 = 0.8)
    3. T2: \{S2, S3\}. 누적: \{S1, S2, S3, S4, S5\}. (5/5 = 1.0)
    4. T3: \{S3, S4\}. 누적: \{S1, S2, S3, S4, S5\}. (5/5 = 1.0)
    - $O_A$ 조기 커버리지 점수: $(0.6 + 0.8 + 1.0 + 1.0) / 4 = 3.4 / 4 = 0.85
  - 순서 $O_B$: \[T2, T3, T4, T1\]
    1. T2: \{S2, S3\}. 누적: \{S2, S3\}. (2/5 = 0.4)
    2. T3: \{S3, S4\}. 누적: \{S2, S3, S4\}. (3/5 = 0.6)
    3. T4: \{S1, S4, S5\}. 누적: \{S1, S2, S3, S4, S5\}. (5/5 = 1.0)
    4. T1: \{S1, S2\}. 누적: \{S1, S2, S3, S4, S5\}. (5/5 = 1.0)
    - $O_B$ 조기 커버리지 점수: $(0.4 + 0.6 + 1.0 + 1.0) / 4 = 3.0 / 4 = 0.75

B. For $O_A$ compute the cumulative unique faults detected after each position and APFD'. Do the same for $O_B$.

- 해석: $O_A$와 $O_B$ 각각에 대해 "각 테스트 실행 후 누적된 고유 결함 탐지 비율의 평균" (APFD')을 계산한다.
- 풀이:
  - 순서 $O_A$: \[T4, T1, T2, T3\]
    1. T4: \{\}. 누적: \{\}. (0/2 = 0.0)
    2. T1: \{\}. 누적: \{\}. (0/2 = 0.0)
    3. T2: \{F1\}. 누적: \{F1\}. (1/2 = 0.5)
    4. T3: \{F2\}. 누적: \{F1, F2\}. (2/2 = 1.0)
    - $O_A$ APFD': $(0.0 + 0.0 + 0.5 + 1.0) / 4 = 1.5 / 4 = 0.375
  - 순서 $O_B$: \[T2, T3, T4, T1\]
    1. T2: \{F1\}. 누적: \{F1\}. (1/2 = 0.5)
    2. T3: \{F2\}. 누적: \{F1, F2\}. (2/2 = 1.0)
    3. T4: \{\}. 누적: \{F1, F2\}. (2/2 = 1.0)
    4. T1: \{\}. 누적: \{F1, F2\}. (2/2 = 1.0)
    - $O_B$ APFD': $(0.5 + 1.0 + 1.0 + 1.0) / 4 = 3.5 / 4 = 0.875

C. Which order is better for coverage? Which is better for fault detection?

- 해석: A와 B의 계산 결과를 비교한다.
- 풀이:
  - 커버리지: $O_A(0.85) > O_B(0.75)$
  - 결함 탐지: $O_B(0.875) > O_A(0.375)$
  - 결론:
    - 커버리지 측면에서는 $O_A$가 더 좋다.
    - 결함 탐지 측면에서는 $O_B$가 더 좋다.

### 5. 강의 개요 (Lec 1: Introduction)

- **소프트웨어 공학(SE)이란?** 소프트웨어의 설계, 개발, 테스트 및 유지보수에 중점을 둔 공학 분야, 신뢰할 수 있는 시스템을 구축하기 위한 체계적인 원칙 기반 방법론을 다룸
- **'공학'이 필요한 이유:** 초기 소프트웨어는 체계적이지 않았으나, 시스템의 규모와 복잡성이 증가하며 비용 초과, 신뢰성 문제 등 '소프트웨어 위기(Software Crisis)'가 대두됨.
- **물리적 시스템과의 차이:** 소프트웨어는 무형적(intangible)이고, 복잡성이 높으며(high complexity), 잦은 변경(frequent changes)이 발생.
- **소프트웨어 공학의 본질적 어려움:** 'No Silver Bullet'에서 언급된 것처럼, 소프트웨어는 본질적으로 복잡성(Complexity), 순응성(Conformity), 변경성(Changeability), 비가시성(Invisibility)이라는 어려움을 가지며, 모든 것을 해결할 만능 해결책(silver-bullet)은 없다.
- **자동화(Automation) 및 최적화(Optimisation)의 필요성:** 현대 소프트웨어는 규모가 방대하고(수백만 LOC), 인간의 수동 작업은 비용이 많이 들고 오류가 발생하기 쉬움. 따라서 SE는 개발 및 유지보수 규모를 확장할 수 있는 프로세스와 도구에 관한 것이며, 이 과정에서 자동화와 최적화가 필수적.

## Metaheuristic
### 1. 메타휴리스틱의 정의

- **휴리스틱 (Heuristics):** 특정 문제를 풀 때 사용하는 "그럭저럭 괜찮은" 실용적 방법이나 지름길을 의미한다. 완벽히 최적화되진 않았으나, 근사치로 충분히 좋은 결과를 제공한다.
- **메타 (Meta):** "넘어서" 또는 "그 자체에 대한"을 의미한다.
- **메타휴리스틱 (Metaheuristics):** 단일 문제의 휴리스틱을 넘어, 다양한 문제에 걸쳐 더 나은 해를 찾도록 휴리스틱을 _안내하는_ **상위 레벨의 일반적인 프레임워크**이다.
    - 이는 본질적으로 **"똑똑한 시행착오(Smart trial and error)"** 전략에 해당한다.
    - 특정 문제에 종속되지 않는 범용성을 갖는다.
    - 근사적(approximate)이며 보통 비결정적(non-deterministic) 특징을 보인다.

### 2. 메타휴리스틱의 핵심 요소

모든 메타휴리스틱 접근법은 다음 4가지 핵심 요소로 구성된다.

1. **표현 (Representation):**
    - "무엇을 시도할 것인가?"에 대한 정의이며, 해(solution)를 인코딩하는 방식이다.
    - _예: 8-Queens 문제의 체스판 상태, 미로 찾기의 이동 순서, 자동 패치 생성에서의 코드 변경 사항._
![](img/Pasted%20image%2020251021174329.png)
2. **연산자 (Operators):**
    - "이전 시도와 어떻게 다르게 할 것인가?"에 대한 정의이며, 현재 해를 수정하여 새로운 후보 해를 생성하는 방법이다.
    - _예: 이동 삽입/삭제/변형, 코드 라인 삭제/수정/교체._
        
3. **적합도 함수 (Fitness Function):**
    - "이번 시도는 얼마나 잘했는가?"를 평가하며, 해의 품질을 정량적으로 측정하는 척도이다.
    - _예: 목표까지의 거리, 통과하는 테스트 케이스의 수._
        
4. **제약 조건 (Constraints):**
    - 반드시 지켜야 할 규칙을 의미한다.
    - _예: 프로그램이 컴파일되어야 함, 최대 이동 횟수._
        

### 3. 주요 개념: 탐험 (Exploration) vs. 활용 (Exploitation)
메타휴리스틱 검색은 항상 두 가지 상반된 힘 사이의 균형을 맞춘다.
- **탐험 (Exploration):**
    - 다양성을 장려하며 검색 공간의 **새로운 영역**을 시도하는 것이다.
    - _지나칠 경우:_ 무작위 검색(random search)이 된다.
- **활용 (Exploitation):**
    - 현재 유망해 보이는 **특정 방향**에 집중하여 최적화하는 것이다.        
    - _지나칠 경우:_ 더 좋은 해를 놓치고 **지역 최적해(local optimal)**에 빠진다.
**핵심 과제:** 이 둘 사이의 균형을 맞추는 것이 모든 최적화 알고리즘의 핵심이다.

### 4. 소프트웨어 공학(SE)에서의 적용과 한계

- **SE 적용 이유:**
    
    - 현대 소프트웨어는 너무 거대하고 복잡하여 수동 작업이 불가능하다.
        
    - 메타휴리스틱은 자동화된 테스트 생성, 프로그램 복구(APR) 등 대규모 SE 문제를 **검색 문제**로 치환하여 실용적인 해결책을 제시한다.
        
- **함정과 도전 과제:**
    
    - **기계의 특성:** 기계는 설정된 적합도 기준(예: 테스트 통과)을 만족하는 **가장 쉬운** 해를 찾는다. 이것이 반드시 _올바른_ 해를 의미하지는 않는다.
        
    - **잘못된 긍정 (False-Positives):** GenProg 사례처럼, 모든 테스트를 통과(Fitness ↑)하지만 실제로는 버그를 고치지 않고 기능을 삭제하는 (semantically wrong) 패치가 생성될 수 있다.
        
    - **구문 (Syntax) vs. 의미 (Semantic):** 대부분의 메타휴리스틱은 코드 구조나 라인 수 같은 **구문적(syntactic)** 표현을 최적화한다. 하지만 '품질', '정확성', '유지보수성'은 **의미론적(semantic)** 개념이며, 구문적 최적화가 항상 의미론적 품질 향상을 보장하지는 않는다.
## Metaheuristics? Heuristic?

![](img/Pasted%20image%2020251021174001.png)
Metaheuristic은 특정 Problem에 종속되지 않았고 Heuristic은 특정 Problem에 종속됨

Heurisitic: 완벽하게 최적화되지 않지만 충분히 좋은 근사치를 제공하는 실용적인 접근 방식, 즉 Short-cut
Meta-heuristic: 더 나은 Solution으로 Guide General Framework를 의미

- Strategies that guide the search of the acceptable solution.
- Approximate and usually non-deterministic
- Not problem specific
- Smart trial and error
  
  교수님께서 예시로 사용하신 퀸 배치 문제
  Brute-force로 구하게 되면 총 좌표 개수에서 퀸을 놓고 하나씩 제거 되어
  $64\cdot 63\cdot 62\cdot 61\cdots 58\cdot 57 = 178462987637760$이다.
  Backtracking으로 Brute-force보다 효율적으로 진행하더라도 시간 복잡도는 $O(n!)$로 아직 높다.
  44Queen 배치 문제 처럼 $N$ 값이 높아지면 높아질수록 이 문제의 난도가 올라가 Metaheuristic을 선택하는 것이 올바르다.

Trial and Guidance를 반복하여 방향성 유지 혹은 새 방향성 찾기
