# 1주차: Introduction - 소프트웨어 개요 및 테스팅 기초

## 1.1 소프트웨어 개요

### 관점의 차이

소프트웨어를 바라보는 관점은 사람마다 다르다. 건축물에 비유하면 시공자, 거주자, 설계사, 작업 인부가 각기 다른 관점을 가지듯이, 소프트웨어도 의뢰자, Designer, Programmer, User 등 다양한 관점이 존재한다.

### 소프트웨어의 정의

- **소프트웨어**: 프로그램과 프로그램의 **개발, 운용, 수정 및 기능 확장**을 위해 필요한 모든 정보
- 소프트웨어는 프로그램의 **동적인 의미**를 포함
- 프로그램은 **형식 언어(Formal Language)**로 표현된 지적 구축물

---

## 1.2 소프트웨어의 특성

| 특성 | 설명 |
|------|------|
| 비가시성 (Invisibility) | 개념적·무형적이며, 구조가 코드 내에 내재되어 있음 |
| 복잡성 (Complexity) | 개발 과정이 복잡하고, 대상 업무 및 시스템이 난해함 |
| 순응성 (Conformity) | 요구나 환경 변화에 적절히 변형 가능 |
| 변경 가능성 (Changeability) | 요구사항 변화에 대응하여 수정이 가능 |
| 복제 가능성 (Duplicability) | 동일한 복제물을 손쉽게 생성 가능 |
| 테스팅의 어려움 (Intestability) | 완벽한 테스트가 현실적으로 불가능 |

---

## 1.3 소프트웨어의 유형

### 소프트웨어 분류 체계

| 분류 기준 | 세부 유형 |
|-----------|-----------|
| **기능** | 시스템 소프트웨어, 응용 소프트웨어 |
| **개발 과정** | Newly-built, 재사용(Reuse), Modification & Adaptation, Porting |
| **요구되는 신뢰도** | Critical ~ Non-critical |
| **소프트웨어 크기** | Programming-in-the-Small ~ Programming-in-the-Large |
| **데이터 특성** | 그래픽, 이미지, 텍스트, 음성 등 |

### 정보시스템 (Information Systems)

- 대량의 데이터 분류, 저장, 검색에 관심
- GUI 기반의 서비스 인터페이스 제공
- **시스템 사례**: 신용카드 승인 서비스, 항공 예약 시스템, 온라인 쇼핑몰
- **특성**:
  - 데이터 설계의 비중이 크고, 제어 구조는 비교적 간단
  - 개발·운용·유지보수에 많은 노력과 비용 소요
  - 문서가 중요한 요소

### 내포 시스템 (Embedded Systems)

- 주 기능이 계산이 아닌 더 큰 시스템에 논리적으로 포함된 시스템
- **특성**: Large scale, Long-lived, Real-time response, Fail-safe reliability, Asynchronous, Highly parallel, Distributed
- **시스템 사례**: 공정 제어(Process Control), 비행기 유도 시스템(Flight Guidance), 교환기 시스템(Switching Systems)

## 1.4 소프트웨어 관련 핵심 질문과 답변

### Q1. 소프트웨어 개발 비용 중 프로그래밍(코딩) 비용의 비중은?

**→ 약 20%**

소프트웨어는 물리적이라기보다 논리적 요소로 구성되므로, 건축(80~90%가 시공)과 달리 코딩은 약 20%만 차지한다.

| 단계 | 비율 |
|------|------|
| 요구분석 · 설계 | 40% |
| 프로그래밍 | 20% |
| 테스팅 | 40% |

### Q2. 배달된 소프트웨어의 실행코드 1,000줄당 예상 오류 수는?

**→ 4개 미만**

- 개발 과정 중 발견 오류: 약 50~60개 / 1,000줄
- 제품 배달 후 발견 오류: 평균 4개 이하

### Q3. 사용자가 발견하기 가장 어려운 오류는?

**→ 제안서와 사용자 요구사항에 대한 잘못된 이해**

- 가장 어려운 문제: 사용자가 무엇을 원하는지 명확하게 정의하는 일
- **핵심 교훈**: "프로그래밍을 서둘러 시작할수록 더 오랜 시간이 걸려 늦게 끝난다."
- 오류의 발견이 늦어질수록 수정 비용이 기하급수적으로 증가

### Q4. 유지보수 비용은 개발 비용의 몇 배?

**→ 약 2배**

- 유지보수 비용은 제품이 체계적으로 만들어졌는지에 **반비례**
- 전체 비용 비율: 개발 비용 33% vs 유지보수 비용 67%

## 1.5 소프트웨어에 대한 오해

### 관리자의 오해

1. 좋은 개발 관련 서적과 표준이 있으면 필요한 모든 것이 충족될 것이다.
2. 최신 장비(Workstation)나 CASE 도구를 도입하면 좋은 제품을 빨리 만들 수 있다.
3. 엔지니어가 요구 분석을 하고 있으면 생산적이지 못한 일을 하고 있다고 생각한다.
4. 공정 지연 시 인력을 더 투입하면 해결된다.

### 고객의 오해

1. 목표에 대한 개략적 기술만으로도 소프트웨어를 만드는 데 충분하다.
2. 사용자 요구사항은 계속 변하며 소프트웨어는 융통성이 있어 쉽게 변경을 수용할 수 있다.

### 엔지니어의 오해

1. 프로그램이 만들어지고 작동하면 임무는 끝난다.
2. 시스템을 작동시켜 보기 전까지는 품질 평가 방법이 없다.
3. 프로젝트의 결과는 작동하는 프로그램뿐이다.

## 1.6 소프트웨어 위기

### 소프트웨어 개발의 문제점

- 예산 초과 (Cost Overruns)
- 개발 일정 지연 (Late Delivery)
- 불충분한 성능 (Inadequate Performance)
- 신뢰하기 어려운 품질 (Unreliability)
- 유지보수의 어려움 (Impossible Maintenance)
- 유지보수 비용의 과다 (Prohibitive Maintenance Costs)

### 소프트웨어 위기 (Software Crisis)

컴퓨터 하드웨어의 급속한 발전과 대중화로 소프트웨어 수요가 급증하였으나, 소프트웨어의 생산성과 생산 기술이 이를 따라가지 못한 현상.

| 항목 | 1965 → 1985 증가율 | 1983 연도 증가율 |
|------|---------------------|-----------------|
| 수요 증가 (Demand) | 100배 | 12% |
| 생산성 (Productivity) | 2배 | 4% |
| 인력 공급 (Manpower) | 10배 | 4% |

## 1.7 소프트웨어 공학

### 정의

> 품질 좋은 소프트웨어를 **최소의 비용**으로 **계획된 일정**에 맞추어 개발하기 위하여 여러 가지 **공학적 원리와 방법을 체계적으로 적용**하는 것

### 소프트웨어 공학의 역사

| 시기 | 주요 사건 |
|------|-----------|
| **~1970** | 소프트웨어 위기 인식, 소프트웨어 공학 탄생 (High-level language programming, Interactive programming, Goto 논쟁(1965)) |
| **~1980** | 소프트웨어 위기 해소책 모색, 학문으로 정착 (Software Lifecycle, Methodology & Tools, Programming-in-the-Large, Information Hiding, Structured Design) |
| **~현재** | 소프트웨어 공학 발전 (SW Environment, SW Reuse, SW Factory, CASE, Design Patterns, SW Components, SW Architecture) |

### 소프트웨어 공학의 이슈

- 저수준의 소프트웨어 생산성 및 품질
- 노동 집약적인 소프트웨어 생산 과정 (가격 상승 원인, 향상된 기술에 둔감)
- 숙련된 소프트웨어 공학자의 부족
- 느린 기술 이전 (Technology Transfer)
- 소프트웨어 생산 과정 자동화, 표준화된 부품을 이용한 생산 필요

### 소프트웨어 공학의 크기 요소

| 구분 | 관심사 |
|------|--------|
| **Programming-in-the-Small** | 비교적 잘 정의된 문제의 구현, 정확한 프로그램 개발, 명세가 변경되지 않음 |
| **Programming-in-the-Large** | Problem definition, Changing requirements, Interface between components, System integration & testing, Configuration management, Maintenance, Documentation |
| **Programming-in-the-Many** | Communication 문제(팀원 간 정보 교환), Work assignment, Quality assurance(표준화), Project management(계획/제어, 팀 조직) |

## 1.8 소프트웨어의 품질 요소

소프트웨어 품질 요소는 크게 **3가지 관점**으로 분류된다 (McCall의 품질 모델 기반):

| 관점 | 품질 요소 |
|------|-----------|
| Product Operation (제품 운용) | Correctness(정확성), Reliability(신뢰성), Efficiency(효율성), Integrity(무결성), Usability(사용성) |
| Product Revision (제품 수정) | Maintainability(유지보수성), Flexibility(유연성), Testability(테스트 용이성) |
| Product Transition (제품 전이) | Portability(이식성), Reusability(재사용성), Interoperability(상호운용성) |

## 1.9 소프트웨어 테스팅 개요

### 구현과 테스팅의 관계

- 프로젝트 규모가 클수록 구현과 개발자 테스트의 **비중이 줄어든다**.
- 프로젝트에서 **구현 오류의 비중이 가장 높다**.

### SW 코드 검증 방법

소스 코드 검증은 크게 두 가지로 나뉜다:

| 구분 | 방법 | 세부 항목 |
|------|------|-----------|
| Analysis (분석) | 소스 코드의 품질 평가 | Language 품질 및 표준, Coding 품질 및 표준 |
| Test (테스트) | 소스 코드의 동작 테스트 | Functional correctness, Temporal correctness, Unexpected behaviour & program violations |

### 소스 코드 검증 기법

| 기법 | 유형 | 설명 |
|------|------|------|
| **Code Inspection** | Static Analysis (정적 분석) | 코드를 실행하지 않고 분석 → 오류 **예방(Prevention)** |
| **Code Execution** | Dynamic Analysis (동적 분석) | 코드를 실행하여 테스트 → 오류 **검출(Detection)** |

### 다양한 테스트 기법들의 오류 감지율

- 어떤 단일 방법도 평균 **75% 이상의 오류 감지율**을 보이지 못함
- 따라서 **다양한 방법을 병행**하여 사용해야 오류 감지율을 높일 수 있음

### 테스팅 생명주기 (V-Model)

V-Model은 개발 단계와 테스트 단계가 대응되는 모델이다:

| 개발 단계 (왼쪽) | 테스트 단계 (오른쪽) |
|-------------------|----------------------|
| Objectives / Requirements | Acceptance Test |
| System Design | System Test / Installation Test |
| Program Structure Design | Function Test |
| Module Interface Specifications | Integration Test |
| Code / External Specification | Module Test |

# 2주차: Project Identification & Management

## 2.1 프로젝트의 시작

### 프로젝트는 어떻게 시작되는가?

- **비즈니스 요구**(Business Needs)가 프로젝트를 이끌어야 한다.
- **프로젝트 스폰서**(Project Sponsor)가 새로운 시스템의 비즈니스 필요성을 인식하고 구현을 원한다.
- 비즈니스 요구가 시스템의 **기능**(Functionality)을 결정한다.
- 프로젝트의 **비즈니스 가치**(Business Value)가 명확해야 한다.

---

## 2.2 System Request

시스템 요청서는 프로젝트의 비즈니스 이유와 기대 가치를 기술한 문서이다.

### System Request의 핵심 요소

| 요소 | 설명 |
|------|------|
| **Project Sponsor** | 프로젝트를 시작하고 비즈니스 측의 주요 연락 담당자 역할을 하는 사람 |
| **Business Need** | 시스템을 시작하는 비즈니스 관련 이유 |
| **Business Requirements** | 시스템이 제공할 비즈니스 역량 |
| **Business Value** | 시스템이 조직에 가져다 줄 이점 |
| **Special Issues / Constraints** | 시스템 구현과 관련된 이슈 및 위원회의 결정 사항 |

### 프로젝트 초기 승인 (Preliminary Project Acceptance)

- 승인 위원회(Approval Committee)가 System Request를 검토
- 제공된 정보를 바탕으로 프로젝트의 가치를 평가
- 가치가 인정된 프로젝트는 승인 후 추가 조사인 **타당성 분석**(Feasibility Analysis)을 진행

## 2.3 타당성 분석 (Feasibility Analysis)

제안된 프로젝트의 기회와 제약을 이해하기 위한 상세한 비즈니스 분석이다. 프로젝트 전반에 걸쳐 지속적으로 재평가된다.

### 2.3.1 기술적 타당성 (Technical Feasibility) — "Can We Build It?"

| 평가 항목 | 세부 내용 |
|-----------|-----------|
| 업무 영역 친숙도 | 사용자와 분석가가 해당 비즈니스 영역에 얼마나 익숙한가 |
| 기술 친숙도 | 해당 기술을 이전에 사용해 본 경험이 있는가 |
| 프로젝트 규모 | 대규모 프로젝트일수록 리스크가 큼 |
| 기존 시스템 호환성 | 기존 시스템과의 호환 가능 여부 |
| **평가 방법** | 이전 프로젝트와 비교, 경험 있는 IT 전문가 자문 |

### 2.3.2 경제적 타당성 (Economic Feasibility) — "Should We Build It?"

경제적 타당성은 4단계로 분석한다:

#### Step 1: 비용과 편익 식별 (Identify Costs and Benefits)

| 구분 | 설명 |
|------|------|
| **유형 비용** (Tangible Costs) | 시스템이 조직에 가져다주는 수익 (예: 매출 증가) |
| **무형 비용** (Intangible Costs) | 직감과 신념에 기반한 비용 (정량화 어려움) |

#### Step 2: 비용과 편익에 값 할당 (Assign Values)

- 어렵지만 필수적인 추정 과정
- 해당 분야에 가장 익숙한 사람들과 협력하여 추정치 개발
- 무형 비용도 가능한 한 정량화
- 정량화 불가 시 목록으로 작성하여 지원 자료에 포함

#### Step 3: 현금 흐름 결정 (Determine Cash Flow)

- **Simple Cash Flow Method** 활용
- 연도별(예: 2026~2030) 현금 유입/유출을 계산하여 총합 산출

#### Step 4: 재무 타당성 평가 (Assess Financial Viability)

**Net Present Value** (NPV, 순현재가치)

```
NPV = PV(미래 현금 유입) - PV(미래 현금 유출)
    = PV(편익) - PV(비용)

PV(현재가치) = 현금 흐름 / (1 + 이자율)^n
```

| n (년) | 6% | 10% | 15% |
|--------|-----|------|------|
| 1 | 0.943 | 0.909 | 0.870 |
| 2 | 0.890 | 0.826 | 0.756 |
| 3 | 0.840 | 0.751 | 0.572 |
| 4 | 0.792 | 0.683 | 0.497 |

- **NPV ≥ 0** → 프로젝트 수용 가능
- **NPV < 0** → 프로젝트 수용 불가

**Return on Investment (ROI, 투자 수익률)**

```
ROI = NPV / PV(현금 유출)
```

**Break Even Point (BEP, 손익 분기점)**

- 프로젝트 투자금을 회수하는 데 걸리는 기간
- 손익분기까지 오래 걸릴수록 프로젝트의 **리스크가 높음**

### 2.3.3 조직적 타당성 (Organizational Feasibility) — "If we build it, will they come?"

| 평가 항목 | 세부 내용 |
|-----------|-----------|
| **조직적 타당성** | 시스템이 사용자에게 수용되고 조직 운영에 통합될 수 있는 정도 |
| **전략적 정합성** (Strategic Alignment) | 프로젝트 목표가 비즈니스 목표와 얼마나 잘 부합하는가 |
| **이해관계자 분석** (Stakeholder Analysis) | 아래 3가지 이해관계자 유형 평가 |

**이해관계자 유형:**

| 유형 | 역할 |
|------|------|
| **Project Champion(s)** | System Request를 생성한 고위 비-IS 경영진 |
| **Organizational Managers** | 새 시스템 사용과 비즈니스 변화 수용을 독려 |
| **System Users** | 개발 전반에 걸쳐 사용자 참여를 촉진해야 함 |

## 2.4 프로젝트 선정 이슈

- 승인 위원회는 System Request와 타당성 분석을 기반으로 작업
- 위원회는 **비즈니스 필요성**과 시스템 구축의 **리스크**를 검토
- **포트폴리오 관리(Portfolio Management)**:
  - 프로젝트가 전체 포트폴리오에 어떻게 부합하는지 평가
  - 균형 잡힌 프로젝트 포트폴리오를 위해 트레이드오프 필요
  - 타당한 프로젝트라도 포트폴리오 이슈로 거절되거나 연기될 수 있음

### 프로젝트 관리의 4가지 핵심 단계

1. **프로젝트 규모 식별** (Identifying Project Size)

3. **프로젝트 인력 배치** (Staffing the Project)

## 2.7 Function Point 추정

### Function Point란?

- 1979년 IBM의 **Allen Albrecht**가 개발한 개념
- 시스템의 **수와 복잡도**를 기반으로 한 프로그램 크기 측정 방법

### Function Point의 5가지 구성 요소

| 구성 요소 | 설명 | 예시 |
|-----------|------|------|
| **Inputs** | 데이터 입력 | 데이터 입력 화면 |
| **Outputs** | 데이터 출력 | 보고서 |
| **Queries** | 데이터 조회 | 데이터베이스 쿼리 |
| **Files** | 내부 파일 | 내부 데이터 파일 |
| **Program Interfaces** | 외부 인터페이스 | 다른 시스템의 파일 |

### Step 1: 시스템 크기 추정

**Function Point Estimation Worksheet 예시:**

| 구분 | 총 수 | Low | Medium | High | 합계 |
|------|-------|------|--------|------|------|
| Inputs | 6 | 3×3 | 2×4 | 1×6 | 23 |
| Outputs | 19 | 4×4 | 10×5 | 5×7 | 101 |
| Queries | 15 | 0×7 | 15×10 | 0×15 | 39 |
| Files | 15 | 0×7 | 15×10 | 0×15 | 150 |
| Program Interfaces | 3 | 1×5 | 0×7 | 2×10 | 25 |
| **TUFP** (Total Unadjusted Function Points) | | | | | **338** |

**Processing Complexity (PC) 요소** (0 = 영향 없음, 3 = 큰 영향):

| 요소 | 값 |
|------|-----|
| Data Communications | 3 |
| Heavy Use Configuration | 0 |
| Transaction Rate | 0 |
| End-User Efficiency | 0 |
| Complex Processing | 0 |
| Installation Ease | 0 |
| Multiple Sites | 0 |
| Performance | 0 |
| Distributed Functions | 2 |
| Online Data Entry | 2 |
| Reusability | 1 |
| Operational Ease | 0 |
| Extensibility | 0 |
| **Total PC** | **8** |

**Adjusted Function Points 계산:**

```
APC (Adjusted Processing Complexity) = 0.65 + (0.01 × PC)
    = 0.65 + (0.01 × 8) = 0.73

TAFP (Total Adjusted Function Points) = APC × TUFP
    = 0.73 × 338 = 247
```

### Function Points → Lines of Code 변환

Function Points는 프로그래밍 언어에 따라 코드 라인 수(LOC)로 변환할 수 있다.

### Step 2: 노력 추정 (Effort Estimation)

- 크기와 생산율의 함수
- **COCOMO 모델** 활용:

```
Effort (Person-Months) = 1.4 × (KLOC)
```

**예시**: LOC = 10,000이면 → Effort = 1.4 × 10 = **14 Person-Months**

### Step 3: 일정 추정 (Schedule Time)

```
Schedule Time (months) = 3.0 × (Person-Months)^(1/3)
```

**예시**:
- Person-Months = 14 → Schedule Time ≈ **7 months**
- Person-Months = 37.42 → Schedule Time ≈ **10 months**

## 2.8 Use Case 추정

### Use Case Diagram 개요

- **Actor**: 외부 에이전트(인간, HW, 다른 시스템)가 수행하는 역할
- **Use Case**: 여러 시나리오를 포함한 완전한 기능 단위

### Use Case Points 추정 절차

#### 1단계: Unadjusted Use-Case Points (UUCP)

| 구분 | 가중치 |
|------|--------|
| **Actor Weighting** | 1 ~ 3 |
| **Use Case Weighting** | 5, 10, 15 |

#### 2단계: Technical Complexity Factor (TCF)

- 13가지 기술 복잡도 요소 평가

```
TCF = 0.6 + (0.01 × TFactor)
```

#### 3단계: Environmental Factor (EF)

- 8가지 환경 요소 평가

```
EF = 1.4 + (-0.03 × EFactor)
```

#### 4단계: Adjusted Use Case Points (UCP)

```
UCP = UUCP × TCF × EF
```

#### 5단계: 노력 추정

```
Effort (Person-Hours) = UCP × 계수
```

**PHM (Person-Hour Factor) 결정 기준:**

| 조건 | 계수 |
|------|------|
| (E1~E6 중 < 3인 개수) + (E7~E8 중 > 3인 개수) ≤ 2 | **20** |
| 위 합계가 3 또는 4 | **28** |
| 그 외 | 프로젝트 재검토 필요 |

## 2.9 작업 계획 수립 및 관리

### 작업 식별 (Identifying Tasks)

| 접근 방식 | 설명 |
|-----------|------|
| **방법론 기반** | 표준 작업 목록 활용 |
| **하향식**(Top-down) | 최상위 작업 식별 → 점점 작은 단위로 분해 → **작업 분해 구조**(WBS) 구성 |

### Project Workplan 구성 요소

| 요소 | 설명 |
|------|------|
| Task Name | 작업 이름 |
| Duration | 작업 기간 |
| Current Status | 현재 진행 상태 |
| Task Dependencies | 작업 간 의존 관계 |
| Milestone (Dates) | 주요 일정 마일스톤 |

### 프로젝트 추적 도구

#### Gantt Chart

- **막대 차트**(Bar Chart) 형식
- 프로젝트 상태를 특정 시점에 모니터링하는 데 유용

#### PERT Chart

- **순서도**(Flowchart) 형식
- 작업 의존관계와 **임계 경로**(Critical Path)를 시각화

### 일정 지연 시 가능한 조치

프로젝트 일정이 지연되었을 때 활용 가능한 전략들이 존재한다.

### 범위 관리 (Managing Scope)

| 개념 | 설명 |
|------|------|
| **Scope Creep** | 새로운 요구사항이 추가되면서 발생하는 일정 및 비용 초과 |
| **Timeboxing** | 고정된 마감일 설정, 필요시 기능을 축소하여 대응 |
