# 1주차: Introduction - 소프트웨어 개요 및 테스팅 기초

## 1.1 소프트웨어 개요

### 관점의 차이

소프트웨어를 바라보는 관점은 사람마다 다르다. 건축물에 비유하면 시공자, 거주자, 설계사, 작업 인부가 각기 다른 관점을 가지듯이, 소프트웨어도 의뢰자, Designer, Programmer, User 등 다양한 관점이 존재한다.

### 소프트웨어의 정의

- **소프트웨어**: 프로그램과 프로그램의 **개발, 운용, 수정 및 기능 확장**을 위해 필요한 모든 정보
- 소프트웨어는 프로그램의 **동적인 의미**를 포함
- 프로그램은 **형식 언어(Formal Language)**로 표현된 지적 구축물

---

## 1.2 소프트웨어의 특성

| 특성 | 설명 |
|------|------|
| **비가시성 (Invisibility)** | 개념적·무형적이며, 구조가 코드 내에 내재되어 있음 |
| **복잡성 (Complexity)** | 개발 과정이 복잡하고, 대상 업무 및 시스템이 난해함 |
| **순응성 (Conformity)** | 요구나 환경 변화에 적절히 변형 가능 |
| **변경 가능성 (Changeability)** | 요구사항 변화에 대응하여 수정이 가능 |
| **복제 가능성 (Duplicability)** | 동일한 복제물을 손쉽게 생성 가능 |
| **테스팅의 어려움 (Intestability)** | 완벽한 테스트가 현실적으로 불가능 |

---

## 1.3 소프트웨어의 유형

### 소프트웨어 분류 체계

| 분류 기준 | 세부 유형 |
|-----------|-----------|
| **기능** | 시스템 소프트웨어, 응용 소프트웨어 |
| **개발 과정** | Newly-built, 재사용(Reuse), Modification & Adaptation, Porting |
| **요구되는 신뢰도** | Critical ~ Non-critical |
| **소프트웨어 크기** | Programming-in-the-Small ~ Programming-in-the-Large |
| **데이터 특성** | 그래픽, 이미지, 텍스트, 음성 등 |

### 정보시스템 (Information Systems)

- 대량의 데이터 분류, 저장, 검색에 관심
- GUI 기반의 서비스 인터페이스 제공
- **시스템 사례**: 신용카드 승인 서비스, 항공 예약 시스템, 온라인 쇼핑몰
- **특성**:
  - 데이터 설계의 비중이 크고, 제어 구조는 비교적 간단
  - 개발·운용·유지보수에 많은 노력과 비용 소요
  - 문서가 중요한 요소

### 내포 시스템 (Embedded Systems)

- 주 기능이 계산이 아닌 더 큰 시스템에 논리적으로 포함된 시스템
- **특성**: Large scale, Long-lived, Real-time response, Fail-safe reliability, Asynchronous, Highly parallel, Distributed
- **시스템 사례**: 공정 제어(Process Control), 비행기 유도 시스템(Flight Guidance), 교환기 시스템(Switching Systems)

## 1.4 소프트웨어 관련 핵심 질문과 답변

### Q1. 소프트웨어 개발 비용 중 프로그래밍(코딩) 비용의 비중은?

**→ 약 20%**

소프트웨어는 물리적이라기보다 논리적 요소로 구성되므로, 건축(80~90%가 시공)과 달리 코딩은 약 20%만 차지한다.

| 단계 | 비율 |
|------|------|
| 요구분석 · 설계 | 40% |
| 프로그래밍 | 20% |
| 테스팅 | 40% |

### Q2. 배달된 소프트웨어의 실행코드 1,000줄당 예상 오류 수는?

**→ 4개 미만**

- 개발 과정 중 발견 오류: 약 50~60개 / 1,000줄
- 제품 배달 후 발견 오류: 평균 4개 이하

### Q3. 사용자가 발견하기 가장 어려운 오류는?

**→ 제안서와 사용자 요구사항에 대한 잘못된 이해**

- 가장 어려운 문제: 사용자가 무엇을 원하는지 명확하게 정의하는 일
- **핵심 교훈**: "프로그래밍을 서둘러 시작할수록 더 오랜 시간이 걸려 늦게 끝난다."
- 오류의 발견이 늦어질수록 수정 비용이 기하급수적으로 증가

### Q4. 유지보수 비용은 개발 비용의 몇 배?

**→ 약 2배**

- 유지보수 비용은 제품이 체계적으로 만들어졌는지에 **반비례**
- 전체 비용 비율: 개발 비용 33% vs 유지보수 비용 67%

## 1.5 소프트웨어에 대한 오해

### 관리자의 오해

1. 좋은 개발 관련 서적과 표준이 있으면 필요한 모든 것이 충족될 것이다.
2. 최신 장비(Workstation)나 CASE 도구를 도입하면 좋은 제품을 빨리 만들 수 있다.
3. 엔지니어가 요구 분석을 하고 있으면 생산적이지 못한 일을 하고 있다고 생각한다.
4. 공정 지연 시 인력을 더 투입하면 해결된다.

### 고객의 오해

1. 목표에 대한 개략적 기술만으로도 소프트웨어를 만드는 데 충분하다.
2. 사용자 요구사항은 계속 변하며 소프트웨어는 융통성이 있어 쉽게 변경을 수용할 수 있다.

### 엔지니어의 오해

1. 프로그램이 만들어지고 작동하면 임무는 끝난다.
2. 시스템을 작동시켜 보기 전까지는 품질 평가 방법이 없다.
3. 프로젝트의 결과는 작동하는 프로그램뿐이다.

## 1.6 소프트웨어 위기

### 소프트웨어 개발의 문제점

- 예산 초과 (Cost Overruns)
- 개발 일정 지연 (Late Delivery)
- 불충분한 성능 (Inadequate Performance)
- 신뢰하기 어려운 품질 (Unreliability)
- 유지보수의 어려움 (Impossible Maintenance)
- 유지보수 비용의 과다 (Prohibitive Maintenance Costs)

### 소프트웨어 위기 (Software Crisis)

컴퓨터 하드웨어의 급속한 발전과 대중화로 소프트웨어 수요가 급증하였으나, 소프트웨어의 생산성과 생산 기술이 이를 따라가지 못한 현상.

| 항목 | 1965 → 1985 증가율 | 1983 연도 증가율 |
|------|---------------------|-----------------|
| 수요 증가 (Demand) | 100배 | 12% |
| 생산성 (Productivity) | 2배 | 4% |
| 인력 공급 (Manpower) | 10배 | 4% |

## 1.7 소프트웨어 공학

### 정의

> 품질 좋은 소프트웨어를 **최소의 비용**으로 **계획된 일정**에 맞추어 개발하기 위하여 여러 가지 **공학적 원리와 방법을 체계적으로 적용**하는 것

### 소프트웨어 공학의 역사

| 시기 | 주요 사건 |
|------|-----------|
| **~1970** | 소프트웨어 위기 인식, 소프트웨어 공학 탄생 (High-level language programming, Interactive programming, Goto 논쟁(1965)) |
| **~1980** | 소프트웨어 위기 해소책 모색, 학문으로 정착 (Software Lifecycle, Methodology & Tools, Programming-in-the-Large, Information Hiding, Structured Design) |
| **~현재** | 소프트웨어 공학 발전 (SW Environment, SW Reuse, SW Factory, CASE, Design Patterns, SW Components, SW Architecture) |

### 소프트웨어 공학의 이슈

- 저수준의 소프트웨어 생산성 및 품질
- 노동 집약적인 소프트웨어 생산 과정 (가격 상승 원인, 향상된 기술에 둔감)
- 숙련된 소프트웨어 공학자의 부족
- 느린 기술 이전 (Technology Transfer)
- 소프트웨어 생산 과정 자동화, 표준화된 부품을 이용한 생산 필요

### 소프트웨어 공학의 크기 요소

| 구분 | 관심사 |
|------|--------|
| **Programming-in-the-Small** | 비교적 잘 정의된 문제의 구현, 정확한 프로그램 개발, 명세가 변경되지 않음 |
| **Programming-in-the-Large** | Problem definition, Changing requirements, Interface between components, System integration & testing, Configuration management, Maintenance, Documentation |
| **Programming-in-the-Many** | Communication 문제(팀원 간 정보 교환), Work assignment, Quality assurance(표준화), Project management(계획/제어, 팀 조직) |

## 1.8 소프트웨어의 품질 요소

소프트웨어 품질 요소는 크게 **3가지 관점**으로 분류된다 (McCall의 품질 모델 기반):

| 관점 | 품질 요소 |
|------|-----------|
| **Product Operation (제품 운용)** | Correctness(정확성), Reliability(신뢰성), Efficiency(효율성), Integrity(무결성), Usability(사용성) |
| **Product Revision (제품 수정)** | Maintainability(유지보수성), Flexibility(유연성), Testability(테스트 용이성) |
| **Product Transition (제품 전이)** | Portability(이식성), Reusability(재사용성), Interoperability(상호운용성) |

## 1.9 소프트웨어 테스팅 개요

### 구현과 테스팅의 관계

- 프로젝트 규모가 클수록 구현과 개발자 테스트의 **비중이 줄어든다**.
- 프로젝트에서 **구현 오류의 비중이 가장 높다**.

### SW 코드 검증 방법

소스 코드 검증은 크게 두 가지로 나뉜다:

| 구분 | 방법 | 세부 항목 |
|------|------|-----------|
| **Analysis (분석)** | 소스 코드의 품질 평가 | Language 품질 및 표준, Coding 품질 및 표준 |
| **Test (테스트)** | 소스 코드의 동작 테스트 | Functional correctness, Temporal correctness, Unexpected behaviour & program violations |

### 소스 코드 검증 기법

| 기법 | 유형 | 설명 |
|------|------|------|
| **Code Inspection** | Static Analysis (정적 분석) | 코드를 실행하지 않고 분석 → 오류 **예방(Prevention)** |
| **Code Execution** | Dynamic Analysis (동적 분석) | 코드를 실행하여 테스트 → 오류 **검출(Detection)** |

### 다양한 테스트 기법들의 오류 감지율

- 어떤 단일 방법도 평균 **75% 이상의 오류 감지율**을 보이지 못함
- 따라서 **다양한 방법을 병행**하여 사용해야 오류 감지율을 높일 수 있음

### 테스팅 생명주기 (V-Model)

V-Model은 개발 단계와 테스트 단계가 대응되는 모델이다:

| 개발 단계 (왼쪽) | 테스트 단계 (오른쪽) |
|-------------------|----------------------|
| Objectives / Requirements | Acceptance Test |

## 2.1 프로젝트의 시작

### 프로젝트는 어떻게 시작되는가?

## 2.2 System Request

## 2.3 타당성 분석 (Feasibility Analysis)

제안된 프로젝트의 기회와 제약을 이해하기 위한 상세한 비즈니스 분석이다. 프로젝트 전반에 걸쳐 지속적으로 재평가된다.

### 2.3.1 기술적 타당성 (Technical Feasibility) — "Can We Build It?"

| 평가 항목 | 세부 내용 |
|-----------|-----------|
| 업무 영역 친숙도 | 사용자와 분석가가 해당 비즈니스 영역에 얼마나 익숙한가 |
| 기술 친숙도 | 해당 기술을 이전에 사용해 본 경험이 있는가 |
| 프로젝트 규모 | 대규모 프로젝트일수록 리스크가 큼 |
| 기존 시스템 호환성 | 기존 시스템과의 호환 가능 여부 |
| **평가 방법** | 이전 프로젝트와 비교, 경험 있는 IT 전문가 자문 |

### 2.3.2 경제적 타당성 (Economic Feasibility) — "Should We Build It?"

경제적 타당성은 4단계로 분석한다:

#### Step 1: 비용과 편익 식별 (Identify Costs and Benefits)

| 구분 | 설명 |
|------|------|
| **유형 비용 (Tangible Costs)** | 시스템이 조직에 가져다주는 수익 (예: 매출 증가) |
| **무형 비용 (Intangible Costs)** | 직감과 신념에 기반한 비용 (정량화 어려움) |

#### Step 2: 비용과 편익에 값 할당 (Assign Values)

- 어렵지만 필수적인 추정 과정
- 해당 분야에 가장 익숙한 사람들과 협력하여 추정치 개발
- 무형 비용도 가능한 한 정량화
- 정량화 불가 시 목록으로 작성하여 지원 자료에 포함

#### Step 3: 현금 흐름 결정 (Determine Cash Flow)

- **Simple Cash Flow Method** 활용
- 연도별(예: 2026~2030) 현금 유입/유출을 계산하여 총합 산출

#### Step 4: 재무 타당성 평가 (Assess Financial Viability)

**Net Present Value (NPV, 순현재가치)**

```
NPV = PV(미래 현금 유입) - PV(미래 현금 유출)
    = PV(편익) - PV(비용)

PV(현재가치) = 현금 흐름 / (1 + 이자율)^n
```

| n (년) | 6% | 10% | 15% |
|--------|-----|------|------|
| 1 | 0.943 | 0.909 | 0.870 |
| 2 | 0.890 | 0.826 | 0.756 |
| 3 | 0.840 | 0.751 | 0.572 |
| 4 | 0.792 | 0.683 | 0.497 |

- **NPV ≥ 0** → 프로젝트 수용 가능
- **NPV < 0** → 프로젝트 수용 불가

**Return on Investment (ROI, 투자 수익률)**

```
ROI = NPV / PV(현금 유출)
```

**Break Even Point (BEP, 손익 분기점)**

- 프로젝트 투자금을 회수하는 데 걸리는 기간
- 손익분기까지 오래 걸릴수록 프로젝트의 **리스크가 높음**

### 2.3.3 조직적 타당성 (Organizational Feasibility) — "If we build it, will they come?"

| 평가 항목 | 세부 내용 |
|-----------|-----------|
| **조직적 타당성** | 시스템이 사용자에게 수용되고 조직 운영에 통합될 수 있는 정도 |
| **전략적 정합성 (Strategic Alignment)** | 프로젝트 목표가 비즈니스 목표와 얼마나 잘 부합하는가 |
| **이해관계자 분석 (Stakeholder Analysis)** | 아래 3가지 이해관계자 유형 평가 |

**이해관계자 유형:**

| 유형 | 역할 |
|------|------|
| **Project Champion(s)** | System Request를 생성한 고위 비-IS 경영진 |
| **Organizational Managers** | 새 시스템 사용과 비즈니스 변화 수용을 독려 |
| **System Users** | 개발 전반에 걸쳐 사용자 참여를 촉진해야 함 |

## 2.7 Function Point 추정

### Function Point란?

- 1979년 IBM의 **Allen Albrecht**가 개발한 개념
- 시스템의 **수와 복잡도**를 기반으로 한 프로그램 크기 측정 방법

### Function Point의 5가지 구성 요소

| 구성 요소 | 설명 | 예시 |
|-----------|------|------|
| **Inputs** | 데이터 입력 | 데이터 입력 화면 |
| **Outputs** | 데이터 출력 | 보고서 |
| **Queries** | 데이터 조회 | 데이터베이스 쿼리 |
| **Files** | 내부 파일 | 내부 데이터 파일 |
| **Program Interfaces** | 외부 인터페이스 | 다른 시스템의 파일 |

### Step 1: 시스템 크기 추정

**Function Point Estimation Worksheet 예시:**

| 구분 | 총 수 | Low | Medium | High | 합계 |
|------|-------|------|--------|------|------|
| Inputs | 6 | 3×3 | 2×4 | 1×6 | 23 |
| Outputs | 19 | 4×4 | 10×5 | 5×7 | 101 |
| Queries | 15 | 0×7 | 15×10 | 0×15 | 39 |
| Files | 15 | 0×7 | 15×10 | 0×15 | 150 |
| Program Interfaces | 3 | 1×5 | 0×7 | 2×10 | 25 |
| **TUFP (Total Unadjusted Function Points)** | | | | | **338** |

**Processing Complexity (PC) 요소** (0 = 영향 없음, 3 = 큰 영향):

| 요소 | 값 |
|------|-----|
| Data Communications | 3 |
| Heavy Use Configuration | 0 |
| Transaction Rate | 0 |
| End-User Efficiency | 0 |
| Complex Processing | 0 |
| Installation Ease | 0 |
| Multiple Sites | 0 |
| Performance | 0 |
| Distributed Functions | 2 |
| Online Data Entry | 2 |
| Reusability | 1 |
| Operational Ease | 0 |
| Extensibility | 0 |
| **Total PC** | **8** |

**Adjusted Function Points 계산:**

```
APC (Adjusted Processing Complexity) = 0.65 + (0.01 × PC)
    = 0.65 + (0.01 × 8) = 0.73

TAFP (Total Adjusted Function Points) = APC × TUFP
    = 0.73 × 338 = 247
```

### Function Points → Lines of Code 변환

Function Points는 프로그래밍 언어에 따라 코드 라인 수(LOC)로 변환할 수 있다.

### Step 2: 노력 추정 (Effort Estimation)

- 크기와 생산율의 함수
- **COCOMO 모델** 활용:

```
Effort (Person-Months) = 1.4 × (KLOC)
```

**예시**: LOC = 10,000이면 → Effort = 1.4 × 10 = **14 Person-Months**

### Step 3: 일정 추정 (Schedule Time)

```
Schedule Time (months) = 3.0 × (Person-Months)^(1/3)
```

**예시**:
- Person-Months = 14 → Schedule Time ≈ **7 months**
- Person-Months = 37.42 → Schedule Time ≈ **10 months**

## 2.8 Use Case 추정

### Use Case Diagram 개요

- **Actor**: 외부 에이전트(인간, HW, 다른 시스템)가 수행하는 역할
- **Use Case**: 여러 시나리오를 포함한 완전한 기능 단위

### Use Case Points 추정 절차

#### 1단계: Unadjusted Use-Case Points (UUCP)

| 구분 | 가중치 |
|------|--------|
| **Actor Weighting** | 1 ~ 3 |
| **Use Case Weighting** | 5, 10, 15 |

#### 2단계: Technical Complexity Factor (TCF)

- 13가지 기술 복잡도 요소 평가

```
TCF = 0.6 + (0.01 × TFactor)
```

#### 3단계: Environmental Factor (EF)

- 8가지 환경 요소 평가

```
EF = 1.4 + (-0.03 × EFactor)
```

#### 4단계: Adjusted Use Case Points (UCP)

```
UCP = UUCP × TCF × EF
```

#### 5단계: 노력 추정

```
Effort (Person-Hours) = UCP × 계수
```

**PHM (Person-Hour Factor) 결정 기준:**

| 조건 | 계수 |
|------|------|
| (E1~E6 중 < 3인 개수) + (E7~E8 중 > 3인 개수) ≤ 2 | **20** |
| 위 합계가 3 또는 4 | **28** |
| 그 외 | 프로젝트 재검토 필요 |

## 3.2 Waterfall 모델

### 기본 구조

Waterfall 모델은 각 단계를 순차적으로 진행하며, 각 단계의 산출물이 다음 단계의 입력이 된다.

| 단계 | 산출물 |
|------|--------|
| User Problem | Statement of Requirements |
| Problem Analysis | Customers Requirements |
| Requirement Specification | Software Requirements Specification |
| Architectural Design | Software Structure |
| Physical Design | Hardware/Software Structure |
| Implementation | Software Modules |
| Test and Debug | Total Software Package |
| Maintain | - |

### 현실적인 Waterfall 모델

실제 프로젝트에서는 이전 단계로의 피드백이 빈번하게 발생한다. 이론적인 순차 진행과 달리, 현실에서는 각 단계 간의 되돌아감이 일어난다.

### Waterfall 모델의 문제점

- 프로세스가 진행되는 동안 변경사항을 수용하기 어려움
- 변화하는 고객 요구사항에 대응하기 어려움
- 프로젝트 수행과정을 명확한 단계로 구분하고 있으나 유연하지 못한 단계 분할 방법
- 요구사항을 잘 이해하고 있는 경우에만 적용 적합
- 이상적 단계 구분에서는 각 단계가 명확히 분리되지만, 실제 단계 분포에서는 활동들이 겹침

---

## 3.3 병렬적 개발 방법론

전체 시스템을 여러 서브프로젝트로 분할하여 병렬적으로 개발하는 방식이다.

- Planning → Analysis → Design 이후, 여러 서브프로젝트(Subproject 1, 2, 3 ...)로 분리
- 각 서브프로젝트가 독립적으로 Design → Implementation 수행
- 모든 서브프로젝트 완료 후 Integration → System으로 통합

## 3.4 프로토타이핑

### 진화적 프로토타이핑 (Evolutionary Prototyping)

- 고객과 협력하여 초기 개발 사양에서 최종 시스템을 발전시키는 것이 목표
- 프로토타입을 수정 및 보완하고 이를 확장하여 시스템을 개발
- 흐름: Planning → Analysis/Design/Implementation 반복 → System Prototype → Implementation → System

### Throwaway 프로토타이핑

- 프로토타이핑의 목표가 시스템 요구사항을 이해하는 것
- 요구사항이 명확해지면 Waterfall 모델로 개발을 진행
- 흐름: Planning → Analysis → Analysis/Design/Implementation 반복 → Design Prototype → Design → Implementation → System

---

## 3.5 점진적 개발

### 점진적 개발 (Incremental Development)

- 시스템을 필요한 일부 기능의 증분(increment)으로 나누어 점진적으로 개발
- 사용자 요구사항에 우선순위를 부여하고, 우선순위가 가장 높은 요구사항을 초기 증분에 포함
- 증분 개발이 시작되면 해당 요구사항은 고정하고, 이후 증분에서 요구사항을 계속 발전/변경시킴
- 각 증분마다 Analysis → Design → Implementation 반복 수행 → System version 1, 2, 3 ... 순차 릴리스

### 장점

- 고객 가치를 각 증분과 함께 제공하여 시스템 기능을 더 빨리 사용할 수 있음
- 초기 증분은 이후 증분에 대한 요구사항을 유도할 수 있는 프로토타입 역할
- 전반적인 프로젝트 실패 위험 감소
- 우선순위가 가장 높은 시스템 서비스는 가장 많은 테스트를 받을 수 있음

### 단점

- 개발이 빠르게 진행되므로 시스템의 매 버전마다 관련 문서를 작성/수정하는 것이 비효율적일 수 있음
- 시스템의 새로운 증분이 추가될 때마다 시스템 구조가 점점 나빠질 수 있음

## 3.6 Unified Process

### 배경

1985~1995년 사이에 50여 가지가 넘는 객체 지향 방법론이 발표되었으며, 가장 성공적인 것으로 Booch Method, Jacobson's Objectory, Rumbaugh의 OMT 등이 있다.

이 세 가지 방법론을 하나로 합친 완벽한 객체 지향 분석 및 설계 방법론이 발표되었다.

| 명칭 변천 | 이름 |
|----------|------|
| 초기 이름 | Rational Unified Process, RUP |
| 변경 이름 | Unified Software Development Process, USDP |
| 현재 이름 | Unified Process, UP |

Unified Process는 적응 가능한(adaptable) 방법론으로, 특정 정보 시스템의 특성에 부합하도록 수정하여 적용해야 한다.

### UP의 Phase와 Workflow

UP는 4개의 Phase와 5개의 Workflow로 구성되며, 각 Phase에서 모든 Workflow가 수행된다(비중은 다름).

| Phase | 주요 활동 비중 |
|-------|-------------|
| Inception Phase | Requirements workflow 비중 높음 |
| Elaboration Phase | Analysis workflow 비중 높음 |
| Construction Phase | Design + Implementation workflow 비중 높음 |
| Transition Phase | Test workflow 비중 높음 |

| Workflow | 설명 |
|----------|------|
| Requirements | 요구사항 수집 및 정의 |
| Analysis | 요구사항 분석 |
| Design | 설계 |
| Implementation | 구현 |
| Test | 테스트 (전체 Phase에서 수행되어야 함) |

## 3.7 애자일 개발

### 애자일 개발 (Agile Development) 개요

모델링 및 문서 작업의 오버헤드와 소요 시간을 상당 부분 줄여서 개발 프로세스를 간소화하는 데 중점을 둔다. 단순하고 반복적인 개발을 강조하며, 대표적인 예로 XP(Extreme Programming)가 있다.

### 애자일 개발의 원리

| 원리 | 설명 |
|------|------|
| 고객 참여 (Customer Involvement) | 고객은 개발 프로세스 전반에 걸쳐 긴밀하게 관여. 새로운 시스템 요구사항 제공, 우선순위 결정, 시스템 반복 평가 |
| 점진적 개발 (Incremental Delivery) | 고객이 각 증분에 포함될 요구사항을 지정하여 증분 단위로 개발 |
| 사람 중심 (People Not Process) | 개발팀의 기술을 인지하고 최대한 활용. 보편적인 개발 절차보다 자신의 업무 방식을 유지 및 개발 |
| 변경 수용 (Embrace Change) | 시스템 요구사항이 변경될 것으로 가정하고 변경사항을 수용하도록 시스템을 설계 |
| 단순성 유지 (Maintain Simplicity) | 소프트웨어와 개발 프로세스 모두에서 단순성에 중점. 복잡성을 제거하기 위해 적극 노력 |

### XP (Extreme Programming)

- 반복적 및 점진적(iterative and incremental) 방법을 이용하는 새로운 접근법
- 프로그래머는 먼저 태스크에 대한 테스트 케이스를 생성하고 자동 테스트 환경을 구성 → 테스트 주도 개발 (Test-Driven Development, TDD)
- 프로그래머는 하나의 컴퓨터에서 파트너와 함께 동시 작업 → 짝 프로그래밍 (Pair Programming)
- 모든 테스트 케이스가 올바르게 작동될 때까지 구현하고 마무리

### XP Release Cycle

Select user stories for this release → Break down stories to tasks → Plan release → Develop/Integrate/Test software → Release software → Evaluate system → (반복)

### Plan-driven vs. Agile

| 구분 | Plan-based Development | Agile Development |
|------|----------------------|-------------------|
| 흐름 | Requirements Engineering → Requirements Specification → Design and Implementation | Requirements Engineering ↔ Design and Implementation (반복) |
| 변경 수용 | Requirements Change Requests를 통해 별도 처리 | 요구사항과 설계/구현이 지속적으로 상호 반복 |

## 3.9 요구사항 분석 개요

### 요구사항 분석의 목표

요구사항 분석은 제안 시스템이 무엇을(What) 수행하는지를 정의하는 것이며, 어떻게(How)는 정의하지 않는다. '어떻게'는 설계 단계에서 정의된다.

흐름: System Requirements → 시스템 요구사항 분석 → (+ Additional Factors/Constraints) → SW 요구사항 명세 → Software Specification

### 요구분석 단계 오류의 중요성

100개의 오류 중에서 요구사항 오류가 차지하는 비중과 수정 비용의 영향이 가장 크다.

| Error Source | 오류 개수 | 수정 비용 | Overall Impact | Percentage Impact |
|-------------|---------|---------|----------------|-------------------|
| Requirements | 56 | 0.82 | 45.92 | 84.8% |
| Design | 27 | 0.13 | 3.51 | 6.5% |
| Coding | 7 | 0.1 | 0.7 | 1.3% |
| Other | 10 | 0.4 | 4.0 | 7.4% |

### 요구사항 분석 프로세스

1. Elicitation (정보 수집): User A, User B, Equipment 등 다양한 정보 소스로부터 정보 수집
2. Specification (정보 기록): 수집한 정보를 요구사항 명세 문서로 기록
3. Analysis (검사 + 피드백): Consistency(일관성), Correctness(정확성), Completeness(완전성)을 검사하고 피드백 반영

---

## 3.10 요구사항 명세의 구성요소

요구사항 명세서는 3가지 요소로 구성된다:

| 구성요소 | 설명 |
|---------|------|
| Functional Requirements (기능적 요구사항) | 시스템이 무엇을 수행하는가 — 행위(Behavior) |
| Non-Functional Requirements (비기능적 요구사항) | 어떤 특성을 가지는가 — 속성(Attribute) |
| Domain Requirements (도메인 요구사항) | 해당 도메인의 기본적인 상식. 인터뷰 과정에서 언급되지 않아 "완전성" 문제 발생 가능 |

### 기능적 요구사항 (Functional Requirements)

시스템이 제공하는 서비스 명세서, 특정 입력에 대해 어떻게 반응해야 하는지, 특정 상황에서 시스템이 어떻게 작동해야 하는지 등의 기능적 요소를 포함한다.

기능적 요구사항의 명세 특성:

- 데이터: 무엇(What), 언제(When), 변형(Transformation) → Data in / Data out
- 기능: 이벤트(Event), 시간(Time)
- 처리: 순차적(Sequential), 병렬적(Parallel), 비정상적 행동(Abnormal Behavior)

기능적 요구사항 예시 — 다목적 전투기 항공전자기기(MRCA AVIONICS):

| 항목 | 내용 |
|------|------|
| System / Subsystem | Flight Control / Autopilot |
| Process | 하드 모드에서 자동조종 장치는 최대 500노트 속도로 고도를 150m에서 ±3m로 제어 |
| Inputs | 고도, 속도 및 지상 속도, 지형 추적 레이더, 공격 각도, Roll/Pitch/Yaw 자이로, 스틱 요구 |
| Outputs | 에일러론(Aileron), 방향타 및 엘리베이터 서보 명령 |
| Major Error | 고도 또는 레이더 신호의 손실 |
| Exception Response | 자동조종 일반 모드, 에너지 비상 모드를 해제함 |

기능적 요구사항 예시 — Insulin Pump:

- 3.2: 필요한 경우, 10분마다 혈당을 측정하고 인슐린을 공급해야 함
- 3.6: 시스템은 매 1분마다 자체 시험 루틴을 수행해야 함

### 비기능적 요구사항 (Non-Functional Requirements)

시스템 속성이나 제약사항 등을 기술한다. 예: 신뢰성, 반응속도, 저장공간 요구사항, I/O 디바이스 용량, 안전성 요구사항 등. 개별 기능이 아닌 전체 시스템에 적용되는 속성들을 기술하며, 기능 요구사항보다 더 중요할 수 있다. 비기능적 요구사항이 충족되지 않으면 시스템이 쓸모없을 수 있다.

비기능적 요구사항의 유형:

| 대분류 | 중분류 | 세분류 |
|--------|--------|--------|
| Product Requirements | Efficiency Requirements | Performance Requirements, Space Requirements |
| Product Requirements | Dependability Requirements | - |
| Product Requirements | Usability Requirements | - |
| Organizational Requirements | Environmental Requirements | - |
| Organizational Requirements | Operational Requirements | - |
| Organizational Requirements | Development Requirements | - |
| Organizational Requirements | Security Requirements | - |
| External Requirements | Regulatory Requirements | Accounting Requirements, Safety/Security Requirements |
| External Requirements | Ethical Requirements | - |
| External Requirements | Legislative Requirements | - |

비기능적 요구사항 예시 — MHC-PMS (Mental HealthCare-Patient Management System):

| 유형 | 내용 |
|------|------|
| Product Requirement | 정상 근무시간(월~금, 08:30~17:30)에 모든 클리닉에서 이용 가능해야 하며, 서비스 불가 시간은 하루 중 5초를 초과하지 않아야 함 |
| Organizational Requirement | 시스템 사용자는 보건당국 신원카드로 인증해야 함 |
| External Requirement | Hstan-03-2006-priv에 설정된 환자 개인정보 보호 조항을 구현해야 함 |

### 검증 가능한 비기능적 요구사항 기술

| 속성 | 측정 방법 |
|------|---------|
| Speed | 초당 처리 트랜잭션 수, 사용자/이벤트 응답 시간, 화면 갱신 시간 |
| Size | Mbytes, ROM 칩 수 |
| Ease of Use | 학습 시간, 도움말 프레임 수 |
| Reliability | 평균 고장 간격(MTTF), 비가용 확률, 고장 발생률 |
| Robustness | 고장 후 재시작 시간, 고장 유발 이벤트 비율, 고장 시 데이터 손상 확률 |
| Portability | 대상 의존 문장 비율, 대상 시스템 수 |

## 3.11 요구사항 명세서 작성법

| 표기법 | 설명 |
|--------|------|
| 자연어 (Natural Language) | 번호가 매겨진 문장으로 작성. 각 문장이 하나의 요구사항 |
| 구조적 자연어 (Structured Natural Language) | 표준 양식 또는 template 형식으로 작성. 각 필드가 요구사항의 측면에 대한 정보 제공 |
| 그래픽 표기법 (Graphical Notations) | 그래픽 모델로 기능 요구사항 정의. UML Use Case Diagram, Class Diagram 등 |
| 수학적 명세 (Mathematical Specifications) | 유한상태기계, 집합 등 수학 개념 기반. 모호성을 줄이지만 고객 이해도 낮음 |

### 자연어 명세

- 다이어그램과 표 등이 포함된 자연어 문장으로 작성
- 표현력이 풍부하고 직관적이며 고객이 쉽게 이해
- 문제점:
  - 명확성 부족: 정확도를 높이면 읽기 어려워짐
  - 요구사항 혼동: 기능적/비기능적 요구사항이 혼재되는 경향
  - 요구사항 합병: 여러 요구사항을 함께 묶어 하나로 표현할 수 있음

### 구조적 명세

- 작성자의 표현 자유를 제한하고 요구사항을 표준 방식으로 작성하도록 함
- 임베디드 제어 시스템에 적합하지만, 정보 시스템에는 너무 엄격할 수 있음

구조적 명세 예시 — Insulin Pump/Control Software/SRS/3.3.2:

| 항목 | 내용 |
|------|------|
| Function | 인슐린 투여량 계산하기 |
| Description | 현재 측정된 당도 수준이 3~7 단위 사이의 안전 영역에 있을 때 투여할 인슐린 용량을 계산 |
| Inputs | 현재 혈당(r2), 이전 혈당값(r0, r1) |
| Source | 센서로부터의 현재 혈당, 메모리에 저장된 이전값 |
| Outputs | CompDose — 인슐린 투여량 |
| Destination | Main Control Loop |
| Action | 혈당 수준이 안정적이거나 떨어지거나 증가율이 감소하면 CompDose = 0. 증가율도 증가하면 현재 혈당과 이전값의 차이를 4로 나눈 값을 반올림. 0으로 반올림되면 최소 용량 설정 |
| Requirements | 혈당 변화율 계산을 위해 이전 값 저장 필요 |
| Pre-condition | 인슐린 저장소에 적어도 일회 최대 허용치의 인슐린 포함 |
| Post-condition | r0 ← r1, r1 ← r2 |
| Side Effects | 없음 |

### UML 기반 시스템 모델링

| 관점 | UML 다이어그램 |
|------|------------|
| Functional Viewpoint | Use Case Diagram |
| Structural Viewpoint | Component Diagram, Package Diagram, Class Diagram, Object Diagram, Deployment Diagram, Composite Structure Diagram |
| Behavioral Viewpoint | State Machine Diagram (Behavioral, Protocol), Activity Diagram |
| Interactions Viewpoint | Sequence Diagram, Communication Diagram, Timing Diagram, Interaction Overview Diagram |

## 3.12 Use Case 분석

### Use Case 분석의 목적

- 고객 요구사항 분석 및 개발 시스템의 범위 정의
- 요구사항을 그래픽 다이어그램 및 시나리오 방식으로 기술
- 고객과 개발자 간의 오해 및 혼동을 최소화
- 시스템 관점에서 외부 상호작용의 적절성 검증
- Acceptance Test의 수행 시나리오로 활용

### Use Case와 Actor

- Use Case: Actor 관점에서 관찰 가능한 연속적인 행위의 집합(a set of sequences of actions)을 기술
- Actor: 시스템과 상호작용하는 외부 에이전트가 수행하는 역할(role). 하나의 외부 에이전트가 여러 역할을 수행 가능 (예: 대출관리자 + 고객)

### Use Case Diagram 구성요소

| 요소 | 표현 방법 |
|------|---------|
| 시스템 | 사각형 박스 |
| Use Case | 사각형 박스 안의 타원 |
| Actor | 시스템 경계 외부의 사람 아이콘 |
| 연관 | Actor와 Use Case 사이의 선 |

### Use Case 설명 기술

Use Case의 내부 시나리오에 대해 기술하며, 초기 설명과 상세 설명으로 나뉜다.

예시 — Set Airfield Altitude:

초기 설명: Navigator는 비행 고도에 대한 새로운 값을 입력한다. Pilot의 헤드 다운 디스플레이에 해당 정보가 표시된다.

상세 설명:
1. Navigator가 비행고도 데이터를 입력한다.
2. 시스템은 데이터의 입력검증을 실시한다.
3. Navigator에게 새로운 값이 디스플레이되고 확인 요청을 한다.
4. Navigator가 확인한다.
5. 새로운 값이 Pilot의 헤드다운 디스플레이에 표시된다.

### Use Case 간의 관계

| 관계 | 표기 | 설명 | 화살표 방향 |
|------|------|------|----------|
| 포함 관계 | \<\<include\>\> | 여러 Use Case들이 공통적인 행위를 하는 경우, 공통 행위를 따로 나타내고 이를 포함. A includes B → A는 B를 내포함 | A → B |
| 확장 관계 | \<\<extend\>\> | 하나의 Use Case의 확장점에서 추가적인 행위를 추가하여 다른 Use Case로 변형. A extends B → A는 B로부터 확장됨 | A → B |

### Use Case 모델링 단계 — 장난감 쇼핑몰 예제

인터넷에서 회원제로 어린이 장난감을 파는 쇼핑몰 구축 시나리오:

| 단계 | 설명 |
|------|------|
| 단계 1: Actor 찾기 | Customer, Shop Manager, Billing System, Shipping Company (도메인 요구사항에서 추출) |
| 단계 2: Use Case 찾기 | Actor별로 Use Case 도출 — Member Registration, Buy Toys, Get Toy Information, Browse Members, Special Customer Management, Add Toy Information to Toy Catalog, Update Toy Catalog 등 |
| 단계 3: 관계 설정 | \<\<extend\>\> (예: Buy Toys → Special Customer Discount), \<\<include\>\> (예: Buy Toys → User Validation, Browse Members → User Validation 등) |
| 단계 4: Use Case 상세 기술 | 선행조건, 기본 흐름, 예외 흐름 기술 |
| 단계 5: Use Case Diagram 검증 | 다이어그램 전체 검토 |

Use Case 상세 기술 예시 — Member Registration:

- 선행조건: 없음
- 기본 흐름:
  1. 고객의 이름, 주소, 주민등록번호, 전화번호, 성별 및 직업 등을 입력
  2. 고객은 자신이 사용할 ID와 패스워드를 입력
  3. 시스템은 고객정보를 저장
- 예외 흐름:
  1. 고객의 ID가 다른 멤버와 중복되면 이 사실을 알리고 다른 ID 사용을 요구

## 3.13 요구사항 불일치의 원인

### 소통의 세 가지 주요 요인

1. 우리가 생각하는 것을 어떻게 말하려는 것으로 바꾸는가?
2. 우리가 어떻게 표현하는지?
3. 우리가 보고 듣는 것을 어떻게 이해 또는 생각하는가?

고객 측: 정보 → 지식 → 문제(Problem) → 요구사항 형성 → 대화(Communication)
개발자 측: 대화(Communication) → 문제 이해/평가 → 디자인 → 해결책

### 요구사항 불일치 발생 원인

| 원인 | 설명 |
|------|------|
| 가정 (Assumption) | 고객의 지식 체계에서 암묵적으로 전제하는 것들 |
| 빠트린 정보 | 인터뷰 과정에서 누락된 정보 |
| 암시적 정보 | 명시적으로 표현되지 않은 정보 |
| 소통 채널 문제 | 대화, 텍스트, 다이어그램 간의 전달 과정에서 발생하는 기술적 문제 |
| 기술적 문제 | 애매성, 모순, 복잡성, 모호함 |
| 지식 트레이드오프 | 고객과 개발자 간 지식 체계의 차이 |

### 요구사항의 단계별 변질 과정

1. 사용자가 자신이 원한다고 생각한 것
2. 사용자가 자신이 원하는 바를 실제로 말로 언급한 것
3. 요구사항 명세서에 기술된 것
4. 개발자가 설계하고 구현한 것
5. 개발된 것을 보고 개발자가 자신이 원했던 것이라고 다시 설명한 것

## 3.14 요구사항 명세서와 테스팅

요구분석 단계에서 요구사항 명세서에 명시된 기능적 및 비기능적 요구사항에 대한 Test Plan을 수립한다. Test Case 작성 과정을 통해 요구사항의 일관성 및 완전성을 검사할 수 있다.

| 입력 | 출력 | 테스트 |
|------|------|--------|
| 요구사항 명세서 (기능적 요구사항 + 비기능적 요구사항) | Test Plan (Test Case + Test Scenario) | System Testing (기능적 테스팅 + 성능 테스팅 등) |
